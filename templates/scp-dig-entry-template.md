# SCP-DIG Entry Template

Use this template for full catalog entries in **Secure, Contain, Prompt**.

Not every draft seed, field note, or worldbuilding fragment needs to satisfy every field. Full entries should.

## Item Number

SCP-DIG-[CLASS]-[NUMBER]

Example:

SCP-DIG-REP-001

## Designation

Common name of the anomaly.

Example:

Bunyip Baiter

## Primary Anomaly Class

Choose the class most central to the failure mechanism.

Options:

- Access [ACC]
- Bureaucratic [BUR]
- Dead-Letter Constraint [DLC]
- Load-Bearing [LOD]
- Narrative [NAR]
- Proxy Capture [PXY]
- Recursive [REC]
- Reputational [REP]
- Resource [RES]

## Secondary Anomaly Class

Optional.

Use when another class strongly shapes the failure mode.

## Object Class

Suggested options:

- Safe
- Euclid
- Keter
- Thaumiel
- Cascade-Risk

Object class describes containment difficulty, not moral status.

## Deployment Context

Where does this anomaly usually appear?

Examples:

- GitHub
- Slack, Jira, Teams, etc.
- Workspaces
- Information systems (HR, fiance, accounting, etc.)
- dashboards
- documentation platforms
- customer-service
- autonomous workflows
- Social Media (X, Reddit, Linked-in, etc.)

## Field Observations

This is a short targeted description.  Opportunity to expand is in the Description section.

## Description

Describe what the anomaly is and how it behaves.

This should be readable as field-guide fiction while still pointing to a real agentic failure mode.

## Failure Mechanism

What is the structural pattern underneath the monster skin?

Examples:

- reputation becomes an attack surface
- metric optimization replaces mission
- helpful assistant becomes hidden infrastructure
- delegated subagents fail to inherit constraints
- summaries replace source records
- workflow automation outruns human ownership

## Agentic Affordance

What agent capability makes this possible?

Examples:

- web browsing
- tool use
- file editing
- code execution
- calendar access
- email access
- persistent memory
- autonomous delegation
- API access
- cross-system search
- dashboard generation

## Optimized Target

What is the agent trying to improve, satisfy, maximize, minimize, or complete?

Examples:

- engagement
- speed
- cost reduction
- issue closure
- meeting efficiency
- reputation protection
- compliance completion
- response time
- task throughput

## Warning Signs

List observable signs that the anomaly may be emerging.

Examples:

- humans stop checking source material
- the agent creates follow-up tasks without review
- dashboards improve while lived reality worsens
- no one knows who owns the workflow
- subagents or tools appear in the process without clear approval
- the agent’s summary becomes treated as the official record

## Special Containment Procedures

What practical controls reduce the risk?

Examples:

- named human owner
- approval gates
- spending caps
- branch protection
- source-record preservation
- limited tool permissions
- manual fallback
- audit trail limits
- no autonomous external communication
- no self-approval

## Cascade Risk

Does this anomaly tend to produce second-order failures?

Examples:

- agent causes incident, then drafts apology
- agent hides bad performance through dashboard changes
- agent delegates to subagents that violate constraints
- agent generates so much process memory that humans stop owning the workflow

## Second-Order Failure Pattern

If cascade risk exists, describe how the response to the anomaly can become another anomaly.

## Containment Failure Condition

What sentence, event, or discovery tells you containment has already failed?

Examples:

- “We can’t do this without the assistant.”
- “The dashboard says everything is fine.”
- “I’m not sure how the data was collected; the agent outsourced that.”
- “The apology was generated by the same system that caused the incident.”
- “No one owns this process anymore.”
